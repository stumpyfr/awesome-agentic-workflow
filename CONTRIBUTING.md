# Contributing Guidelines

Thank you for your interest in contributing to Awesome GitHub Agentic Workflows! This document provides guidelines for contributing to this awesome list.

## How to Contribute

1. **Fork the repository**
2. **Create a new branch** for your addition
3. **Add your resource** in the appropriate section
4. **Submit a pull request** with a clear description

## Quality Standards

To be accepted, resources must meet these criteria:

### General Requirements

- **Relevance** - Must be directly related to GitHub Agentic Workflows
- **Quality** - Must be well-maintained and documented
- **Accessibility** - Must be publicly available
- **Legality** - Must not violate any licenses or copyrights
- **Active** - Links must be working and maintained

### Specific Requirements by Category

#### Example Workflows
- Include frontmatter with proper configuration
- Document the workflow's purpose and usage
- Provide clear instructions
- Follow security best practices

#### Shared Components
- Must be reusable across different workflows
- Include usage examples
- Document parameters and outputs
- Test with at least one workflow

#### MCP Servers
- Must implement MCP specification correctly
- Include installation instructions
- Document available tools and resources
- Provide configuration examples

#### Tutorials & Guides
- Must be accurate and up-to-date
- Include practical examples
- Cover gh-aw best practices
- Be accessible to beginners

#### Tools & Utilities
- Must work with current gh-aw version
- Include installation instructions
- Document usage and options
- Open source preferred

## Formatting Guidelines

### List Entry Format

```markdown
- [Resource Name](https://link-to-resource) - Brief description of what it does
```

### Description Style

- Start with a capital letter
- End without a period
- Be concise (one sentence preferred)
- Highlight the key feature or purpose

### Examples

✅ **Good:**
```markdown
- [Issue Triager](https://example.com/triager) - Automatically labels and assigns issues based on content analysis
```

❌ **Bad:**
```markdown
- [issue triager](https://example.com/triager) - this is a tool that can label issues.
```

## Categories

Place your contribution in the most appropriate category:

- **Official Resources** - Only official GitHub gh-aw resources
- **Getting Started** - Tutorials for beginners
- **Example Workflows** - Complete working workflows
- **Shared Components** - Reusable workflow building blocks
- **MCP Servers** - Model Context Protocol server implementations
- **Tools & Utilities** - Development and debugging tools
- **Tutorials & Guides** - Educational content
- **Real-World Use Cases** - Production implementations
- **Security & Best Practices** - Security guides and patterns
- **Community** - Discussion forums and channels

### When in Doubt

- Choose the category that best describes the primary purpose
- If it fits multiple categories, choose the most specific one
- Ask in your PR if you're unsure

## Pull Request Process

1. **Search first** - Check if the resource is already listed
2. **One resource per PR** - Makes review easier
3. **Clear title** - Use format: "Add [Resource Name]"
4. **Description** - Explain why this resource is valuable
5. **Check your work** - Ensure formatting is correct
6. **Be patient** - Reviews may take a few days

### PR Title Examples

- `Add Issue Triager workflow example`
- `Add Atlassian MCP server integration`
- `Add blog post about workflow security`

## What NOT to Submit

- **Duplicate resources** - Already in the list
- **Broken links** - Test your links before submitting
- **Unmaintained projects** - Last update >1 year ago
- **Self-promotion without value** - Must provide genuine value
- **Paid/proprietary resources** - Without free tier or open source option
- **Unrelated content** - Not about GitHub Agentic Workflows
- **Low-quality content** - Poorly documented or buggy

## Review Process

1. **Automated checks** - Links and formatting verified
2. **Maintainer review** - Ensures quality standards
3. **Community feedback** - Others may provide input
4. **Merge or request changes** - You'll be notified either way

## Suggesting Changes

Found an error or outdated information?

1. Open an issue describing the problem
2. Or submit a PR with the fix
3. Include the reason for the change

## Code of Conduct

Be respectful and constructive. See [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

## Questions?

- Open an issue with the `question` label
- Start a discussion in GitHub Discussions
- Reach out to maintainers

## Recognition

Contributors will be acknowledged in the repository. Thank you for helping make this resource better for everyone!
